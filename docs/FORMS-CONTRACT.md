# Контракт обычных форм

## Назначение

Этот документ фиксирует проектный контракт для обычных форм сайта.

Он используется как источник правды для:

- клиентского скрипта `public/js/forms.js`;
- разметки форм в компонентах;
- Google Apps Script;
- структуры Google Таблицы.

## Границы контракта

### Входит в контракт

- popup-форма;
- lead-формы;
- любые будущие обычные формы заявок, которые должны писать в общую таблицу.

### Не входит в контракт

- referral форма;
- `src/components/partners/PartnerApplySlot.astro`;
- referral endpoint;
- referral таблица.

Новый единый скрипт не должен обрабатывать referral форму ни при каких условиях.

## Базовый принцип

Обычная форма может содержать любой допустимый поднабор канонических полей.

Контракт строится не вокруг фиксированного состава полей, а вокруг:

- единого словаря имен;
- единого формата метаданных;
- единого payload;
- единых правил инициализации формы.

## Канонические пользовательские поля

Разрешенные пользовательские поля:

- `name`
- `phone`
- `telegram`
- `email`
- `message`

### Правила

- если поле есть в форме, оно должно использовать только каноническое имя;
- старые имена вроде `fullName` и `full_name` в новых формах не используются;
- если поле отсутствует в форме, оно не должно добавляться искусственно.

## Канонические поля согласий

Разрешенные поля согласий:

- `consent_privacy`
- `consent_personal_data`
- `consent_marketing`

### Формат

На фронте согласия отправляются как:

- `"true"` если чекбокс отмечен;
- поле отсутствует или пусто, если чекбокс не был включен и не входит в форму.

## Идентификация формы

Каждая обычная форма должна передавать:

- `form_type`
- `form_id`

### `form_type`

Назначение:

- группировка форм по бизнес-смыслу;
- маршрутизация логики в Apps Script при необходимости;
- аналитика по типам заявок.

Рекомендуемые значения:

- `popup`
- `lead`
- `callback`
- `contact`

### `form_id`

Назначение:

- уникальная идентификация конкретной формы на странице или в проекте.

Примеры:

- `lead-popup-main`
- `homepage-lead`
- `academy-lead`
- `allcourses-lead`

## Служебные метаданные

Каждая заявка должна иметь следующие системные поля:

- `page`
- `page_title`
- `referrer`
- `session_id`
- `submission_id`
- `ga_client_id`

### Правила

- `submission_id` генерируется на клиенте для каждой отправки;
- `session_id` хранится и переиспользуется в пределах сессии/локального хранилища;
- `page`, `page_title`, `referrer` собираются автоматически клиентским скриптом.

## Атрибуция

Контракт поддерживает две группы UTM-данных:

### Last touch

- `lt_source`
- `lt_medium`
- `lt_campaign`
- `lt_content`
- `lt_term`

### First touch

- `ft_source`
- `ft_medium`
- `ft_campaign`
- `ft_content`
- `ft_term`

### Правила

- если данных нет, поля остаются пустыми;
- фронт собирает данные автоматически;
- форма не должна руками дублировать эти поля обычными input-полями.

## Данные триггера

Если форма открыта или инициирована кнопкой, должны поддерживаться поля:

- `trigger_id`
- `trigger_text`
- `trigger_source`
- `trigger_variant`
- `trigger_campaign`

### Назначение

- понимать, какая кнопка или CTA привела к отправке;
- различать несколько точек входа на одной странице.

### Правила

- если триггера нет, поля остаются пустыми;
- если форма открыта из popup-кнопки, данные триггера собираются автоматически.

## reCAPTCHA

Контракт поддерживает reCAPTCHA как опциональную часть формы.

Служебные поля:

- `recaptcha_token`
- `recaptcha_score`
- `recaptcha_success`

### Правила

- `recaptcha_token` используется только в момент отправки;
- в таблицу пишутся `recaptcha_score` и `recaptcha_success`;
- обязательность reCAPTCHA задается конфигурацией формы.

## Формат payload

### Основной принцип

Payload должен быть плоским на верхнем уровне.

Это упрощает:

- разбор в Apps Script;
- маппинг в колонки таблицы;
- отладку;
- поддержку вариативных форм.

### Обязательные верхнеуровневые поля

- `form_type`
- `form_id`
- `submission_id`

### Пользовательские поля верхнего уровня

- `name`
- `phone`
- `telegram`
- `email`
- `message`

### Служебные поля верхнего уровня

- `page`
- `page_title`
- `referrer`
- `session_id`
- `ga_client_id`
- `trigger_id`
- `trigger_text`
- `trigger_source`
- `trigger_variant`
- `trigger_campaign`
- `lt_source`
- `lt_medium`
- `lt_campaign`
- `lt_content`
- `lt_term`
- `ft_source`
- `ft_medium`
- `ft_campaign`
- `ft_content`
- `ft_term`
- `recaptcha_token`

### Полный JSON-слепок

Поле `raw_json` в таблице должно содержать сериализованный payload целиком.

## Правила валидации

### Глобальное минимальное правило

Любая обычная заявка валидна, если заполнен хотя бы один контактный канал:

- `phone`
- `telegram`
- `email`

### Локальные правила формы

Форма может задавать собственные ограничения поверх глобального правила.

Поддерживаемые механизмы:

- HTML `required`
- `data-required-any`
- `data-required-all`

### Интерпретация

#### `data-required-any`

Форма считается валидной, если заполнено хотя бы одно поле из перечисленных.

Пример:

- `data-required-any="phone,telegram"`

#### `data-required-all`

Форма считается валидной, если заполнены все перечисленные поля.

Пример:

- `data-required-all="name,phone"`

### Приоритет

Приоритет правил:

1. встроенная HTML-валидация;
2. `data-required-all`;
3. `data-required-any`;
4. глобальное минимальное правило контакта.

## Контракт data-атрибутов формы

Каждая обычная форма должна иметь:

- `data-form`
- `data-form-type`
- `data-form-id`

### Обязательные `data-*`

- `data-form`
- `data-form-type`
- `data-form-id`

### Поддерживаемые опциональные `data-*`

- `data-success-message`
- `data-success-redirect`
- `data-success-delay`
- `data-required-any`
- `data-required-all`
- `data-recaptcha`

## Значение data-атрибутов

### `data-form`

Признак того, что форма входит в единый обычный контур.

Без этого атрибута `forms.js` форму игнорирует.

### `data-form-type`

Тип формы.

Пример:

- `popup`
- `lead`
- `callback`

### `data-form-id`

Уникальный идентификатор формы.

### `data-success-message`

Пользовательский текст успеха.

Если не задан, используется дефолтный текст скрипта.

### `data-success-redirect`

URL для перехода после успешной отправки.

### `data-success-delay`

Задержка редиректа в миллисекундах.

### `data-required-any`

CSV-список полей, из которых должно быть заполнено хотя бы одно.

### `data-required-all`

CSV-список полей, которые должны быть заполнены все.

### `data-recaptcha`

Флаг, нужен ли токен reCAPTCHA для этой формы.

Допустимые значения:

- `true`
- `false`

## Контракт UI-элементов

Чтобы единый скрипт мог работать с разными формами, у формы должны быть предсказуемые точки интерфейса.

### Поддерживаемые элементы

- submit-кнопка;
- контейнер статуса;
- контейнер debug, если он нужен.

### Рекомендуемые атрибуты

- `data-form-status`
- `data-form-debug`

### Правила

- если контейнер статуса отсутствует, скрипт не должен падать;
- если debug-блок отсутствует, скрипт не должен падать;
- submit-кнопка должна определяться стандартно через `button[type="submit"]`.

## Правила нормализации данных

Перед отправкой фронт должен:

- применять `trim()` к строковым полям;
- удалять пустые строки;
- не отправлять лишние ключи;
- не переименовывать канонические поля;
- не отправлять дублирующие алиасы.

### Telegram

Рекомендуемое правило:

- хранить без ведущего `@`, если это удобно для всей системы;
- либо хранить как ввел пользователь, но тогда правило должно быть единым для всех форм.

Для проекта рекомендуется хранить `telegram` без `@`.

### Phone

Рекомендуемое правило:

- хранить нормализованное строковое значение;
- не пытаться на фронте агрессивно преобразовывать в формат, который может исказить номер.

## Контракт ответа Apps Script

Единый Apps Script должен возвращать JSON.

Минимальный формат:

```json
{
  "ok": true
}
```

### При успехе

```json
{
  "ok": true,
  "row": 123
}
```

### При дедупликации

```json
{
  "ok": true,
  "dedup": true
}
```

### При ошибке

```json
{
  "ok": false,
  "error": "message"
}
```

## Решения, зафиксированные для проекта

На текущем этапе считаем утвержденным:

1. Обычные формы используют только канонические имена полей.
2. Referral форма в единый контракт не входит.
3. Контактный минимум: хотя бы одно из `phone`, `telegram`, `email`.
4. `form_type` и `form_id` обязательны для всех обычных форм.
5. Payload проектируется плоским.
6. Полный слепок заявки хранится в `raw_json`.
7. Новый фронтовый скрипт и новый Apps Script строятся именно по этому документу.
