# План перехода на единый скрипт отправки заявок

## Цель

Полностью переписать с нуля контур обычных форм сайта:

- новый единый клиентский скрипт;
- новый единый Google Apps Script;
- новая единая структура Google Таблицы;
- поддержка вариативных форм с неполным набором полей;
- реферальная форма остается отдельной и в эту миграцию не входит.

Переход допускается без сохранения обратной совместимости со старой схемой полей и старым Apps Script.

## Что рефакторим

### Входит в единый контур

- popup-форма;
- lead-формы на страницах;
- любые будущие обычные формы заявок;
- все формы, которые должны попадать в общую таблицу заявок.

### Не входит в единый контур

- `src/components/partners/PartnerApplySlot.astro`;
- отдельный referral endpoint;
- отдельная referral таблица;
- текущая referral логика.

Это жесткое ограничение проекта.

## Исходная проблема

Сейчас обычные формы работают через несколько разных реализаций:

- `public/js/popup.js`;
- `public/js/leadcapture.js` (removed);
- разные имена полей;
- разная логика сборки payload;
- дублирование UTM и метаданных;
- дублирование UX-логики отправки.

Из-за этого сложно:

- добавлять новые типы форм;
- делать формы с неполным набором полей;
- поддерживать единый формат записи в таблицу;
- централизованно менять логику отправки и валидации.

## Ключевые требования

### 1. Без backend

Backend не добавляется.

Все обычные формы отправляются напрямую в Google Apps Script.

### 2. Скрипт и таблица проектируются заново

Старый Apps Script можно не сохранять.

Допускается:

- новый `HEADERS`;
- новый `doPost(e)`;
- новый формат payload;
- новая таблица;
- новая логика валидации.

### 3. Поддержка вариативных форм

Единый механизм должен принимать формы с разным набором полей.

Допустимые примеры:

- только `telegram`;
- только `phone`;
- `name + phone`;
- `name + telegram`;
- `name + phone + telegram`;
- `name + phone + message`;
- `email + message`;
- любой другой допустимый поднабор канонических полей.

### 4. Обязательность задается локально

Нет единого глобального обязательного набора полей для всех форм.

Каждая конкретная форма сама определяет:

- какие поля у нее есть;
- какие из них `required`;
- нужен ли редирект;
- нужен ли отдельный success message.

### 5. Единый словарь полей

Все обычные формы должны использовать один словарь имен.

Старые расхождения типа:

- `fullName`
- `full_name`
- `name`

в новой схеме устраняются.

## Правило валидности заявки

### Главный принцип

Форма может быть неполной, но заявка должна содержать хотя бы один контактный канал.

### Минимальное правило приема

Заявка считается валидной, если заполнено хотя бы одно из полей:

- `phone`
- `telegram`
- `email`

Поле `name` не должно быть универсально обязательным для всех форм.

### Дополнительные ограничения

Если отдельная форма требует более строгую валидацию, это задается:

- через HTML `required`;
- через `data-required-any`;
- через `data-required-all`.

Но это должно быть локальным правилом формы, а не глобальным правилом всего сайта.

## Новый Apps Script

### Задача

Новый Apps Script должен быть рассчитан именно на вариативные формы.

Он не должен предполагать, что у каждой формы всегда есть:

- `name`;
- `phone`;
- `telegram`.

### Ожидаемое поведение

Скрипт должен:

- принимать JSON или `x-www-form-urlencoded`;
- понимать единый payload;
- безопасно работать с отсутствующими полями;
- записывать пустые значения в отсутствующие колонки;
- валидировать минимум один контактный канал;
- при необходимости валидировать reCAPTCHA;
- защищаться от дублей через `submission_id`;
- сохранять весь объект в `raw_json`.

### Рекомендуемая логика Apps Script

1. Распарсить запрос.
2. Нормализовать объект.
3. Проверить `submission_id`.
4. Проверить наличие хотя бы одного контакта.
5. Проверить reCAPTCHA, если она включена для этой формы.
6. Собрать строку в порядке нового `HEADERS`.
7. Записать строку в таблицу.
8. Вернуть JSON-ответ.

### Рекомендация по reCAPTCHA

В новой версии лучше сделать reCAPTCHA не жестко обязательной для каждой формы, а управляемой:

- либо включать ее для всех обычных форм;
- либо управлять через `form_type`;
- либо передавать `requires_recaptcha` из фронта.

Если оставить ее обязательной всегда, это нужно прямо зафиксировать как общее правило.

## Новый клиентский скрипт

### Файл

- `public/js/forms.js`

### Основная задача

Один скрипт должен обслуживать все обычные формы сайта.

### Что делает `forms.js`

1. Находит формы с `data-form`.
2. Пропускает все, что не относится к обычным формам.
3. Собирает поля формы.
4. Удаляет пустые значения.
5. Применяет единый словарь имен полей.
6. Добавляет `form_type` и `form_id`.
7. Добавляет атрибуцию.
8. Добавляет `submission_id`.
9. Добавляет `raw_json`.
10. Отправляет на единый Apps Script endpoint.
11. Управляет UI отправки.
12. Делает success redirect при настройке.

### Чего `forms.js` делать не должен

- не должен обрабатывать referral форму;
- не должен содержать ветки под popup и lead как под разные системы;
- не должен требовать фиксированного набора полей;
- не должен знать бизнес-логику конкретной страницы.

## Рекомендуемый декларативный контракт формы

Для обычных форм:

- `data-form`
- `data-form-type="lead"`
- `data-form-id="main-lead"`
- `data-success-message="Заявка отправлена"`
- `data-success-redirect="/thank-you"`
- `data-success-delay="1500"`
- `data-required-any="phone,telegram,email"`
- `data-recaptcha="true"`

### Примеры

#### Форма только с Telegram

- поле `telegram`
- `data-required-any="telegram"`

#### Форма name + phone

- поля `name`, `phone`
- `data-required-all="name,phone"`

#### Общая lead-форма

- поля `name`, `phone`, `telegram`
- `data-required-any="phone,telegram"`

## Риски

### 1. Случайное подключение referral формы

Нужно жестко ограничить селектор нового скрипта, чтобы он не обрабатывал referral блок.

### 2. Разные требования к reCAPTCHA

Если часть форм должна работать без reCAPTCHA, это надо заранее заложить в контракт, иначе новый GAS снова станет слишком жестким.

### 3. Избыточно сложный payload

Не стоит делать слишком сложную вложенную структуру.

Лучше:

- плоские основные поля в колонках;
- полный JSON-слепок в `raw_json`.

### 4. Смешивание старой и новой схемы

Если решено переписывать с нуля, не стоит одновременно поддерживать несколько промежуточных форматов дольше, чем нужно.

## Критерии завершения

Проект можно считать завершенным, когда:

1. Popup-форма работает через новый `forms.js`.
2. Lead-формы работают через новый `forms.js`.
3. Все обычные заявки попадают в новую единую таблицу.
4. Формы с неполным набором полей успешно отправляются.
5. Referral форма не затронута.
6. `leadcapture.js` удален, `popup.js` используется только для управления модалкой.

## Итоговый принцип

Новая система должна строиться не вокруг фиксированного набора полей, а вокруг единого механизма отправки, который умеет принимать любой допустимый поднабор канонических полей обычной формы.

Referral контур остается полностью отдельным.
